Documentation

Local-first documentation indexing and retrieval for coding agents.

Introduction

plshelp crawls documentation websites, ingests local Markdown and text files, cleans the content, stores it locally, and gives you a query interface that works well for coding agents. The goal is simple: scrape once, index locally, and stop depending on remote docs search during development.

The tool is local-first by design. It stores pages, chunks, embeddings, and library metadata in SQLite, keeps exported artifacts as plain .md and .txt files, and exposes both human-readable terminal output and machine-readable --json responses. Query-time retrieval can use BM25, embeddings, or hybrid scoring.

What plshelp is good at

Use it when you want stable local context for docs-heavy workflows, code assistants, personal notes, framework references, or merged corpora that would otherwise be spread across many sites or files.

Installation

The supported install path today is the hosted shell installer for Unix-like systems and the PowerShell installer for Windows. Both download a prebuilt release binary and place it in a user bin directory.

macOS and Linux

shell 
curl -fsSL https://plshelp.run/install.sh | sh

Windows (PowerShell)

powershell 
irm https://plshelp.run/install.ps1 | iex

Verify the install

shell 
plshelp help

If the binary is installed successfully but the shell cannot find plshelp, add the installer’s target directory to your PATH and retry.

Quickstart

The fastest way to start is a two-command workflow: crawl and index a docs site with add, then query it locally.

shell 
plshelp add nextjs https://nextjs.org/docs
plshelp query nextjs "how does app router work"

If you already crawled a library, use index to rebuild chunking and embeddings. If you are indexing a local file instead of a URL, skip add entirely and use index --file.

Shell note

Quote questions in the shell, especially if they contain characters like ? or *. The CLI accepts unquoted multi-word queries in many cases, but quoting is still the safest and most portable form.

Docs sites

Use add when your source is a documentation website. It is the high-level command that runs the full pipeline: crawl, chunk, embed, and store the library locally.

shell
plshelp add rust https://doc.rust-lang.org/book/

If you want the crawl output saved as plain files while indexing, add --include-artifacts. With no path, artifacts are written into the library’s compiled directory. You can also pass an explicit output path.

crawlCrawl a docs site and store pages without chunking or embedding. Use this when you want to separate the pipeline into steps.
addRun the full URL-based pipeline in one command. This is the default path for documentation sites.
--include-artifactsAlso emit docs.md and docs.txt during URL-based workflows.

Local files

Use index --file when your source is already local. This is the clean path for personal notes, team playbooks, exported docs, or any hand-written Markdown and text you want to query like a library.

shell
plshelp index mynotes --file ./notes/rust.md
plshelp query mynotes "what did i write about ownership"

The same flag shape also works with chunk if you want to stop before embeddings and inspect the intermediate state. For local files, index is generally the right wrapper unless you explicitly want the pipeline stages separated.

Querying

query searches a single library or merged group. ask searches across multiple libraries. trace shows ranking internals and is the right tool when you need to debug retrieval quality.

shell
plshelp query nextjs "data fetching in app router" --mode hybrid --top-k 3 --context 1
plshelp trace nextjs "data fetching in app router" --mode hybrid --top-k 3 --context 1
plshelp ask "what is broadcasting" --libraries numpyuser,numpy-ref --json
--modeChoose hybrid, vector, or keyword.
--top-kControl how many results are returned. The default is 1.
--contextInclude neighboring parent chunks with each result. The default is 0.
--librariesFor ask, provide a comma-separated library list. Without it, ask searches all available libraries.
alias formYou can also query directly with <library_name> "<question>".

Merged libraries

merge builds a new logical library from existing libraries. This is the right pattern when you want one query target spanning several docs sets, such as a framework guide plus its API reference.

shell
plshelp merge numpy numpyuser numpy-ref
plshelp query numpy "what is broadcasting"

Use --replace if the merged library already exists and you want to overwrite its membership. Merged libraries can also be exported like any other library.

Exports

export writes the compiled library text back out as plain files. This is useful when you want a cleaned docs snapshot, a shareable artifact, or source material for another system.

shell
plshelp export numpyuser
plshelp export numpyuser ./out/numpyuser

By default, exports land in the configured artifacts directory under the library name and use the fixed filenames docs.md and docs.txt.

Inspection and maintenance

The rest of the CLI is designed to make the local corpus inspectable and scriptable. These commands are useful once the initial indexing flow is working and you need visibility into what is stored.

listList all libraries and merged groups.
show <library>Print rollup stats such as page count, parent count, child chunk count, embedding progress, and BM25 coverage.
open <chunk_id>Inspect a specific child chunk and its parent context.
alias <library> <alias>Add an alternate lookup name for a library.
refreshRecompute and backfill stats without crawling again.
configPrint the active config.toml path and contents. Use --json for tooling.
initCreate or update AGENTS.md and CLAUDE.md in the current directory. Use --agents, --claude, or --print as needed.
chunkRun only the chunking and BM25 indexing stage.
embedRun only the embedding stage for a library that is already chunked.

Removal

remove deletes a library or merged group from the local database. Use it when a crawl is stale, a source URL changed, or you want to reclaim local storage before rebuilding.

shell
plshelp remove nextjs

Removal affects the SQLite corpus only. Exported artifacts on disk are not automatically deleted unless you remove them yourself.

Pipeline model

The command surface is intentionally composable. add is the URL wrapper over crawl plus indexing. index is the indexing wrapper over chunking plus embedding. crawl, chunk, and embed also exist as individual stages when you want control or debugging visibility.

initWrite or print repo-local AGENTS.md and CLAUDE.md guidance for agent workflows.
addURL source, full pipeline: crawl → chunk/BM25 → embed.
crawlURL source, crawl only. Populates pages and optional artifacts but stops before chunking.
indexLibrary or file source, chunk + embed. The right wrapper for local files.
chunkChunking + BM25 only. Useful when you want immediate lexical search before embeddings.
embedEmbedding only. Assumes chunked child chunks already exist.

Read commands

Read-side commands are the ones agents will use most heavily. They are all local and can all be made machine-readable with --json where noted.

queryRetrieve from one library or merged group. Returns parent content and source URL.

askCross-library retrieval. Can search all libraries or a comma-separated subset with --libraries.
listEnumerate libraries and merged groups.
showInspect one library’s rollups, chunk counts, embedding counts, and model metadata.
openOpen one child chunk by id and show its parent context.
configPrint the active config path and contents so you can verify runtime settings.

Data model

Internally, plshelp stores crawled pages, parent chunks, and child chunks separately. Parents are the coherent units returned to users and agents. Children are the smaller units used for BM25 indexing and embeddings. Querying ranks child chunks, deduplicates by parent, and then returns the parent content with source metadata.

librariesOne row per library, including rollup stats and current model metadata.
pagesRaw cleaned page-level content captured by crawl.
parentsHeading-aware or size-bounded parent chunks returned by query.
chunksChild chunks used for BM25 and vector retrieval, each linked to one parent.
chunks_ftsSQLite FTS5 index over child chunk content for BM25-style lexical search.
library_aliasesAlternate names for lookup convenience.
library_groupsMerged library membership.
jobsLong-running command history and status output.

Artifacts and output

plshelp always keeps the database as the canonical indexed corpus, but the file outputs matter too. URL-based commands can optionally write artifacts during crawl. Export writes them on demand. The registry builder writes them as static public outputs.

docs.mdCompiled Markdown-like export of a library or registry source.
docs.txtPlain-text export, currently identical in content shape to the markdown artifact.
default export pathartifacts/<library>/docs.md and artifacts/<library>/docs.txt.
registry path/registry/docs/<slug>/docs.md and /registry/docs/<slug>/docs.txt.

BM25

BM25 gives plshelp an exact-text retrieval path that works immediately after chunking. This is important for documentation, where exact API names, flags, error strings, and identifiers often matter more than pure semantic similarity.

In practice, BM25 is the fastest way to make a newly chunked library queryable. It also serves as the lexical half of hybrid retrieval.

Embeddings

Embeddings are generated locally and stored in SQLite as child-chunk vectors. The current default model is AllMiniLML6V2Q, which is a speed-oriented local choice. Parent chunks are kept for display coherence while child chunks are used for ranking.

Query-time embeddings are generated on the fly from the question text. If a library has not been embedded yet, keyword mode still works, and hybrid mode can fall back toward the lexical path.

Hybrid search

Hybrid mode blends vector similarity with BM25. This is usually the right default for technical docs because it captures both semantic intent and exact identifier matches. In plshelp, child chunks are ranked and then deduplicated back to their parent chunks for output.

That means retrieval stays granular, while the final returned content remains coherent enough to feed into an agent prompt.

Scoring and defaults

The current retrieval defaults are intentionally conservative: hybrid mode, top-k = 1, context window = 0, and a weighting split of 90% vector to 10% BM25. These are runtime-configurable and are read fresh from config.toml on every invocation.

default modehybrid
default top-k1
default context0
hybrid weights0.90 vector, 0.10 BM25

JSON output

Every read path and most write paths support --json. This is the integration surface for automation and agents. Success responses are structured JSON objects, and when --json is present, command failures are also emitted as JSON.

shell
plshelp query rust "how do generics work" --json
plshelp show rust --json
plshelp add nextjs https://nextjs.org/docs --json

Codex

Codex can use plshelp by shelling out to the CLI and consuming --json responses. The main useful commands are query, trace, show, list, and open.

If you want agent prompts to stay deterministic, prefer explicit library names and explicit retrieval flags instead of relying on shell aliases or implicit defaults.

Claude Code

Claude Code can use the same pattern: invoke plshelp locally, request JSON, and feed the returned parent content and source URLs into the next step. The value here is that the docs retrieval stays local and reproducible instead of depending on whatever web search happens to surface.

MCP and wrappers

plshelp does not currently ship with a built-in MCP server. The supported integration surface today is the CLI itself. If you want MCP, the practical approach is to wrap the JSON commands in a thin local server or adapter layer.

Practical advice

Build wrappers after the core CLI behavior is stable. The current command surface is already sufficient for most terminal-based agent integrations.

Agent setup and init

plshelp init is the lowest-friction way to expose plshelp to terminal coding agents. It writes repo-local instruction files for agent workflows directly into the current directory and keeps the plshelp section updateable with marker blocks instead of blindly overwriting the full file.

By default, plshelp init writes both AGENTS.md and CLAUDE.md. Use --agents or --claude if you only want one file, and use --print when you want to inspect the generated content without writing anything to disk.

shell
plshelp init
plshelp init --agents
plshelp init --claude
plshelp init --print
plshelp init --print --json

What gets written

The generated files tell agents to prefer plshelp ... --json for local documentation retrieval, to inspect indexing state with show before assuming a query is wrong, and to use trace only when ranking or retrieval quality needs debugging. The embedded templates also include a setup block for brand-new repos:

generated guidance
### Setup (if no libraries are indexed yet)
- plshelp add <name> <docs-url> to index a library
- plshelp show <name> --json to confirm it's ready before querying

How to use this in practice

Run plshelp init once at the project root after you have a library indexed, commit the generated files if you want the instructions to travel with the repo, and then let agents call query, trace, show, list, open, and config through JSON mode. This keeps retrieval local, deterministic, and inspectable.

Config file

plshelp writes a default config.toml on first run and reloads it every time a new command starts. Missing or malformed values fall back to built-in defaults, so the config file is an override layer rather than a fragile required dependency.

shell
plshelp config
plshelp config --json
plshelp init
plshelp init --claude
plshelp init --print
toml
[embedding]
model = "AllMiniLML6V2Q"
batch_size = 128

[chunking]
parent_min_chars = 1400
parent_max_chars = 3000
child_min_chars = 400
child_max_chars = 800
child_split_window_chars = 50

[retrieval]
default_mode = "hybrid"
default_top_k = 1
default_context_window = 0
hybrid_vector_weight = 0.9
hybrid_bm25_weight = 0.1

[sqlite]
journal_mode = "WAL"
busy_timeout_ms = 5000

Storage layout

By default, plshelp stores all of its runtime state in your local application data directory. On macOS this is typically ~/Library/Application Support/plshelp, on Linux it is usually ~/.local/share/plshelp or the XDG data directory, and on Windows it would normally live under %APPDATA%\plshelp.

That directory contains the SQLite database, exported artifacts, embedding model cache, and the generated config.toml. The installed binary itself lives separately in a bin directory so it can be placed on your shell PATH.

Models and chunking

The default embedding model is AllMiniLML6V2Q. Current chunking defaults are tuned around that model’s practical input size: parents default to 1400–3000 characters, while child chunks default to 400–800 characters with a 50-character whitespace search window.

Parent chunks preserve output coherence. Child chunks are what get embedded and ranked. This split keeps query results readable while still using finer-grained retrieval internally.

FAQ

What can I index?

You can index documentation websites, local Markdown files, local text files, and merged collections built from multiple libraries.

Can I query before embeddings finish?

Yes. BM25 is built during chunking, so keyword retrieval works immediately after that stage.

Where does data live?

The binary is installed into a user bin directory. Runtime state lives in the application data directory and includes SQLite, artifacts, model cache, and config.

Can I use this with coding agents?

Yes. That is one of the main intended workflows. Use --json and treat plshelp as a local retrieval subprocess.

Does plshelp require a hosted vector database?

No. Retrieval is fully local.

Current status

The current build supports local config, JSON output, BM25 plus embedding-based retrieval, local file indexing, merged libraries, exports, and a separate registry builder. The install scripts and release workflow are in place, but ecosystem packaging beyond direct installers is still evolving.

The documentation should be treated as the current implementation guide, not a long-term compatibility promise. If you are integrating this into automation, pin a release and test the JSON output you depend on.
